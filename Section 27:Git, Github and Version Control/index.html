1. Introduction to Version Control & Git

Foundations

Version control tracks changes to files so you can access any snapshot in the past, compare revisions, and collaborate without overwriting each other's work. It is a safety net and a collaboration tool rolled into one.

Git is the most popular distributed version control system. Every developer has the full history locally, so you can work offline and synchronize later. Start new projects with the following commands:

  • git init — create a repository in the current folder.
  • git status — inspect tracked/untracked files.
  • git add <file> — stage work for the next snapshot.
  • git commit -m "message" — record the staged snapshot.
  • git log --oneline — review project history.

2. GitHub & Remote Repositories

Collaboration

A remote is a hosted copy of your repository. GitHub provides tooling for issues, pull requests, and social coding on top of Git. Typical remote workflow:

  1. git remote add origin <url> — link your local repo to the hosted one.
  2. git push -u origin main — publish local commits to the remote branch and set upstream tracking.
  3. git pull (or git fetch + git merge) — bring down teammates' changes.

Keep local and remote histories aligned to avoid diverging work and painful conflicts.

3. Managing a .gitignore

Housekeeping

Use .gitignore to exclude generated artifacts, secrets, or anything that does not belong in version control. Patterns can target files, directories, or extensions. 

# Ignore dependencies
node_modules/

# Ignore build output
dist/

# Ignore local config
*.log
*.local.json

Commit the .gitignore itself so that every collaborator benefits from the same clean history.

4. Cloning Existing Projects

Getting Started

Cloning downloads an entire repository, including history, branches, and tags. This is how you begin collaborating on an existing codebase. 

git clone https://github.com/user/project.git
cd project
git status

After cloning, configure remotes if needed and create your own branches to make changes safely.

5. Branching & Merging

Parallel Work

Branches let you isolate work. The main/master branch remains stable while you experiment elsewhere.

  • git branch feature/login — create a branch pointer.
  • git switch feature/login or git checkout feature/login — move to that branch.
  • git merge feature/login — integrate finished work back into the target branch.

Resolve merge conflicts by editing the files, staging the fixes, and committing. Consider rebasing for a linear history when collaborating closely.

6. Forking & Pull Requests

Open Source

Forking creates your own remote copy of someone else's repository. You can then propose changes via a pull request (PR) without write access to the original repo.

  1. Fork on GitHub, then git clone your fork locally.
  2. Add the original repo as upstream with git remote add upstream <url>.
  3. Create a feature branch, commit, and git push origin feature-branch.
  4. Open a PR comparing your branch to the original project's branch. Provide context and request review.

Keep forks updated by fetching from upstream and rebasing or merging regularly.
